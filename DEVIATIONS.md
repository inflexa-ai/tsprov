# DEVIATIONS

Intentional, documented divergences from the reference Python `prov` (v2.1.1,
`reference/prov/`). Each is harmless to PROV-DM semantics or is forced by a
TypeScript/JS constraint. Anchors point into `reference/prov/src/prov`.

> Process: log every deliberate divergence here with its source anchor, the
> reason, and the blast radius. See `docs/migration/02-migration-roadmap.md` §4.

| # | Area | Python behavior (anchor) | tsprov behavior | Reason | Impact |
|---|------|--------------------------|-----------------|--------|--------|
| D1 | `Namespace.qname()` | Constructs a **fresh** `QualifiedName` each call (`identifier.py:160-161`). | Routes through the memoized `qn()`, returning the cached instance. | Interning is the whole point of `intern.ts`; value-equality is identical. | None — values are `equals()`-equal; the TS version just shares instances. |
| D2 | Datetime sub-second precision | `dateutil` + `datetime` keep **microseconds** (6 digits); `isoformat()` emits them. | luxon has **millisecond** resolution, so `…123456` round-trips as `…123000` (`datetime.ts`). | luxon is the chosen datetime lib (`03 §2`); no pure-JS lib preserves µs cleanly + offset. | Byte-parity breaks only for fixtures with sub-ms µs that aren't ms-multiples. **M5 must carry the raw lexical string for such literals.** |
| D3 | Naive (offset-less) datetimes | Kept naive; `isoformat()` emits no offset. | luxon assigns a zone (`fromISO`/`fromJSDate`), so an offset appears (`datetime.ts`). | luxon has no naive-datetime concept. | PROV data is normally tz-aware; revisit at M5 if the corpus has naive times. |
| D4 | Datetime parse leniency | `dateutil.parser.parse` accepts many non-ISO forms. | luxon `fromISO` is **ISO-8601 only** (`datetime.ts`). | Stricter is safer; xsd:dateTime is ISO. | Authoring with a non-ISO string throws where Python coerced. |
| D5 | `Literal` value stringification | `str(value)`: `str(2.0)=="2.0"`, `str(True)=="True"` (`model.py:155`). | `String(value)`: `String(2.0)==="2"`, `String(true)==="true"` (`literal.ts`). | JS `number` can't distinguish `2` from `2.0`; `String` is the idiom. | Harmless for the corpus **iff the JSON decoder passes lexical strings** (it must, M5). Only bare-number authoring is affected — the int/double collapse (risk register). |
| D6 | XSD numeric parsing | Bare `int()`/`float()` **raise** on malformed input (`model.py:98-114`). | `XSD_DATATYPE_PARSERS` numeric parsers return `null` (`literal.ts`). | Returning `null` is safer than throwing mid-parse. | Corpus values are always well-formed; only pathological input differs. |
| D7 | Warnings | `ProvWarning(Warning)` class + `logger.warning`/`debug` (`model.py:235`, `:158`, `:167`). | A settable `WarningHandler` callback (`setWarningHandler`, default `console.warn`) in `error.ts`. | No TS `Warning` analogue; an injectable callback is testable and library-friendly (`03 §3`). | Behaviorally equivalent (a diagnostic message); callers can silence or collect. |
| D8 | `ProvActivity.setTime` | Stores the time value **raw** (un-coerced) and replaces the set (`model.py:786,802`). | Same — `setTime` stores raw (a string stays verbatim; a JS `Date` → `DateTime` since `Date ∉ AttrValue`) and replaces via `AttributeStore.set` (`element.ts`). | Faithful to the Python quirk; raw strings also avoid the D3 naive-offset issue, *helping* byte-parity. | None / positive — a naive ISO string round-trips byte-identical. |
| D9 | `AttributeStore` ordering & read-mutation | `defaultdict(set)`: unordered values; reads insert empty keys (`model.py:293`). | Insertion-ordered, `valueKey`-deduped maps; reads never create entries (`record/attributes.ts`). | JS has no value-set; ordering is deterministic and read-mutation is a Python footgun. | TS output is deterministic where Python's set order was not (PROV equality is order-independent). |
| D10 | `unified()` | Shares the source's `NamespaceManager` by reference (`model.py:2603`). | Same — `ProvDocument.unified` assigns `doc._namespaces = this._namespaces` (shared). | Faithful to the Python quirk. | Mutating one document's namespaces would affect its unified copy. |
| D11 | `ProvDocument.flattened` | Returns `self` (identity) when there are no bundles (`model.py:2593`). | Same — returns `this` unchanged. | Faithful to the Python quirk. | A "flattened" document may be identity-equal to the original. |
| D12 | `add_attributes` formal-attribute conflict | A second, *different* value for a single-valued formal attribute **always** raises `ProvException` (`model.py:505-524`), so `unified()` (`model.py:1681`) crashes on any such clash. | `unified()`/`addAttributes` gain an **opt-in** `formalAttributeConflict` policy: `"throw"` (default, byte-identical to Python), `"first"` (keep earliest-recorded value), `"last"` (last-write-wins), both by record insertion order (`record/record.ts`, `bundle.ts`). | Replayed-observation event sources (e.g. a source that re-emits a record on crash recovery) need a deterministic merge instead of a serialization-time crash. | Default path is byte-identical to Python (still throws); the merge policies are strictly opt-in. |
| D13 | `prov_to_graph` container scope | Converts only document-level records: `prov_document.unified()` then `unified.get_records(...)`, so records inside bundles are **invisible** to the graph (`graph.py:68`). Its round-trip test skips bundled documents entirely (`test_graphs.py:11-13`). | `ProvGraph.of` builds from `document.flattened().unified(options)`, hoisting bundle records to the document level first, so **bundled records participate** as nodes/edges (`src/graph/graph.ts`). | The reverse-adjacency substrate this feeds (lineage) must index provenance regardless of which bundle a record was authored in; a graph blind to bundle contents would silently drop those relations. | `graph.document` (and any `graphToProv` round-trip) is the flattened, unified transform — bundle structure is intentionally collapsed, so a round-tripped document has no sub-bundles. Because `flattened()` hoists every bundle's records to one identifier-keyed `unified()` scope, a cross-bundle **same-identifier collision** now merges bug-for-bug as Python's document-level `unified()` does (`model.py:1649-1668`): records sharing an identifier merge into the first, so a relation that shares an identifier with an element union-merges INTO that element and never reaches edge extraction — invisible even to `skipped` (which only sees relations that survived the merge). |
| D14 | Inferred-endpoint sentinel | An endpoint referenced but never declared becomes a synthetic element built with a **`bundle=None`** sentinel (`graph.py:82-84`); `graph_to_prov` skips any node whose `bundle is not None` is false (`graph.py:103`). | A `ProvRecord` always requires a `RecordBundle` resolver (`record.ts:165-175`), so the null-bundle sentinel cannot port. The synthetic element is instead built against the transformed document as its **resolver only** (never registered — the constructor does not register, `bundle.ts:389-435`), and the node carries an explicit **`inferred: boolean`** flag; `graphToProv` skips `inferred` nodes (`src/graph/graph.ts`). | The type system cannot express a record with no owning bundle; an explicit discriminant is honest where Python used an out-of-band `None`. | Behaviorally identical to Python — inferred nodes never round-trip back into a document; the document is never mutated by inference. |
| D15 | Render scene: under-endpoint relations & blank nodes (`prov_to_dot`) | A relation whose endpoint is unset is drawn **to a synthetic blank node** (`_get_bnode`, `dot.py:301-306,313`); one with fewer than two endpoint slots is dropped with a **silent `continue`** (`dot.py:354-355`). | `toRenderScene` emits **no blank nodes** (a DOT presentation trick, excluded this stage) and, when the first two formal endpoints are not both resolvable, omits the edge and **records it in `scene.skipped`** with a reason (`packages/tsprov-render-core/src/scene.ts`). | The loop forbids silent drops, so dot.py's silent `continue` is made observable; blank-node materialization is stage-2 presentation, not semantic scene data. | The scene draws strictly fewer edges than `prov_to_dot` — a single-endpoint relation (e.g. `wasAssociatedWith(a1)`) becomes an enumerable `skipped` entry instead of a dangling blank-node edge. 30 corpus fixtures + 0 real-world docs exercise this. |
| D16 | Render scene traversal substrate | `prov_to_dot` walks the **document model**, recursing into sub-bundles as clusters (`dot.py:328-330`) and drawing n-ary legs from `formal_attributes` (`dot.py:338-352`). | `toRenderScene` mirrors that walk over the document model; it deliberately does **not** reuse `tsprov/graph` (`scene.ts`). | `ProvGraph` flattens sub-bundles away (see D13) and binarizes relations, discarding the bundle structure and n-ary legs a faithful scene needs; extending it for rendering would be a core change (hard rule 1). This corrects `rendering.research.md`'s graph-view direction. | None vs Python — the scene is faithful to `prov_to_dot`. The divergence is from the internal research doc, not the reference; `ProvGraph` stays the lineage tool. |
| D17 | DOT annotation row order (`DotRenderer`) | `prov_to_dot` re-orders a note box's attribute rows with `sorted_attributes` — the PROV-XML order (formal attrs, then `label`/`location`/`role`/`type`/`value`, then the rest alphabetically) (`dot.py:224`, `model.py:2799`). | `DotRenderer` emits annotation rows in the scene's order, which is the record's **insertion** order (`toRenderScene` preserves it and calls sorting "a renderer concern"; `packages/tsprov-render-dot/src/dot.ts`). | The scene carries only stringified `{name, value, uris}` per attribute, not the record-type formal-attribute tables `sorted_attributes` needs; re-deriving that PROV-XML order in the renderer would pull tsprov record internals across the package boundary for a purely cosmetic re-ordering. | Per-row **content** is byte-identical to Python (same escaping, hrefs, text); only the **order** of rows within one note box can differ. No structural effect — the golden comparator matches rows as a set (the design's tolerated "attribute order" noise). |
| D18 | DOT blank-node routing for unset tail slots (`prov_to_dot`) | `prov_to_dot` decides blank-node routing by **slot count**: `add_nary_elements = len(nodes) > 2 and show_nary` (`dot.py:352`) counts a relation class's formal QName endpoint slots whether or not they are set, so a >2-slot relation is split through a point-shaped **blank node even when every tail slot is unset** — the unset legs are then dropped at `dot.py:379-382`'s `if node is not None`. So `wasAssociatedWith(activity, agent)` with no plan renders as `activity → bnode → agent` with a mid-edge dot and no legs. | `toRenderScene` emits n-ary legs only for **set** tail endpoints (`value instanceof QualifiedName`, `scene.ts`); a relation with two resolvable endpoints and no set tails carries no legs, so `DotRenderer`'s `needsBlankNode = naryLegs.length > 0 || attributes.length > 0` (`packages/tsprov-render-dot/src/dot.ts`) is false and it draws a **single direct edge — no blank node**. | The blank node in that case carries **zero information** (no legs, no annotation) — a presentation artifact of Python's slot-counting, not PROV semantics; the direct edge conveys the same meaning more cleanly. The golden-parity comparator reconstructs relations across the optional blank node (`dot-extract.ts`), so both forms compare equal. | DOT output differs from `prov_to_dot` for every >2-slot relation with unset tails (visually: no mid-edge point node). When a relation DOES have a set tail or an annotation, `needsBlankNode` is true and routing matches Python exactly. |
| D19 | Mermaid shape approximations (`MermaidRenderer`) | `prov.dot` draws the PROV visual language with Graphviz shapes: entity `oval`, activity `box`, agent **`house`**, bundle **`folder`** (`DOT_PROV_STYLE`, `dot.py:102-115`), the n-ary/annotation join as a **`shape=point`** dot (`dot.py:304`), and the attribute annotation as a **`shape=note`** HTML-TABLE box (`dot.py:212-247`). Graphviz colours include names like **`red4`** (`prov:Usage`, `dot.py:120`). | `MermaidRenderer` maps onto Mermaid flowchart primitives, which lack `house`/`folder`/`point`/`note`: entity → stadium `([…])`, activity → rect `[…]`, agent → **hexagon `{{…}}`**, an inferred bundle endpoint → **subroutine `[[…]]`**, the join node → a **tiny gray circle `((" "))`** (near-invisible `bnode` classDef), and the annotation → a **gray rect `[…]`** with `<br/>`-joined `name = value` rows (`packages/tsprov-render-mermaid/src/mermaid.ts`). Theme colours stay Graphviz-faithful — `PROV_THEME` is the single source of truth — while the emitter owns the CSS surface: a Graphviz-only colour name that is not valid CSS (currently only `red4`) is projected to its X11 hex (`red4` → `#8B0000`) by `toCssColor` at emission time, so the `classDef`/`linkStyle` colour survives the browser's CSS parser instead of being dropped. | Mermaid has no Python reference and a fixed shape vocabulary; these are the honest best-fit glyphs for the same PROV convention. Blank-node routing itself is byte-for-byte D18-consistent — the shape of the join node differs, not when it appears. A tiny emitter-owned projection table (`toCssColor`, currently the single entry `red4` → `#8B0000`) keeps `PROV_THEME` Graphviz-faithful while the CSS the emitter writes stays browser-legal — the styling contract is still theme-conformant AND the colour actually reaches the browser. | Mermaid output is visually close but not identical to `prov.dot`: agents are hexagons not houses, bundles subroutines not folders, the join a small circle not a point, notes plain rects; `prov:Usage` edges render dark red in a browser (`red4` is projected to `#8B0000`); every other theme colour is already valid CSS. Structure — which nodes/edges exist, when a relation splits — is unchanged from the DOT renderer. Goldens are our own reviewed-once snapshots (`packages/evals/goldens/mermaid/`), not a Python oracle. |
| D20 | SVG layout & bundle grouping (`SvgRenderer`) | `prov.dot` emits a DOT graph whose pixel geometry is produced by **Graphviz's own layout engine** (`dot`) once the string is piped to it, and sub-bundles are real Graphviz **`cluster` subgraphs** for which the engine reserves contiguous space (`dot.py:249-257`). | `SvgRenderer` owns layout itself, running **dagre's layered algorithm** (`@dagrejs/dagre`) over the scene with node boxes sized from an over-approximating text-metric estimator, then serialising glyphs/edges straight to an SVG string — so coordinates are dagre's, NOT Graphviz `dot`'s. dagre has no native clusters, so each bundle is drawn as a **post-hoc bounding rectangle** computed from its already-placed member boxes and painted behind them (`packages/tsprov-render-svg/src/svg.ts`), not a layout-reserved region. | There is no Python SVG reference — `prov.dot` produces DOT, and the picture is Graphviz's; reproducing Graphviz's coordinates is the stretch **graphviz stage**'s job and is an explicit Non-Goal here (design Goals/Non-Goals). The visual CONTRACT is `PROV_THEME` — shapes, fills, strokes, labels — which IS faithful (SVG even restores the real `house`/`folder`/`note`/`point` glyphs Mermaid could not); only geometry differs. dagre is synchronous and deterministic, keeping `render()` sync and the goldens platform-stable. | Layout/coordinates differ from `prov.dot` + Graphviz by design — no pixel parity (that is the graphviz stage). Because a bundle box is computed post-hoc, in a pathological graph a member node can spatially interleave with non-members (accepted this stage; dagre `compound` mode is a noted `TODO(extend)` in the emitter). Which nodes/edges exist, the theme colours/shapes, and D18 blank-node routing (marker-less first segment) are unchanged from the DOT/Mermaid renderers. Goldens are our own reviewed-once snapshots (`packages/evals/goldens/svg/`), not a Python oracle. |
| D21 | Link-scheme allowlisting (all renderers) | `prov_to_dot` emits an identifier's URI into a live link **verbatim**, with no scheme check: the node `URL='"%s"' % uri` (`dot.py:279`) and the annotation row `' href="%s"' % value.uri` when the value is an `Identifier` (`dot.py:232`). It inherits pydot/Graphviz, which do not filter link schemes. | Every tsprov renderer routes each emitted link through render-core's `safeLinkUri` allowlist — `http`/`https`/`mailto`, or a scheme-less (relative/fragment/protocol-relative) reference — and drops the link for any other scheme: the SVG `<a href>`, Mermaid's `click … href`, DOT's node/cluster `URL` **and** the HTML-`TABLE` annotation-row `href`s, and the interactive payload URIs plus the client's panel/attribute anchors (`packages/tsprov-render-core/src/uri.ts`). Separately, DOT's `useLabels` two-line HTML-like node label entity-escapes its interpolated label/QName text (`&`→`&amp;`, `<`→`&lt;`, `>`→`&gt;`; `packages/tsprov-render-dot/src/dot.ts` `nodeLabel`), whereas the reference interpolates it raw (`dot.py:269-273` — its annotation rows `html.escape` but this label does not); an unescaped `>` would otherwise close the `<…>` label early and forge arbitrary DOT, including a `URL="javascript:…"` that never passes through `safeLinkUri`, so this is an extra hardening step closing the same sink. | A PROV identifier URI is `namespace.uri + localpart` — attacker-influenceable document content, not trusted authoring — and the renderers' intended workflow is a single self-contained file opened straight from `file://`. A `javascript:`/`data:` URI in an `href`/`URL` (the DOT one goes live once Graphviz rasterizes to SVG) would execute script in that origin the instant a reader clicks it. | A hostile-scheme URI renders as inert text (interactive/DOT value cell) or simply no link (SVG anchor, Mermaid click, DOT node `URL`) instead of a clickable link; the display text is never lost. Every real corpus URI is absolute `http(s)`, so **no golden contains a non-allowlisted scheme** and every DOT/SVG/Mermaid golden is byte-unchanged (only the interactive payload goldens moved, for an unrelated field trim). The hostile case is proven by `packages/evals/link-scheme.test.ts`, deliberately kept OUTSIDE the curated golden set so a reviewed regeneration can never bless one. The `useLabels` label escape is likewise byte-transparent: real labels/QNames carry no `&<>`, so `uselabels.gv` and every DOT golden stay unchanged, while a hostile `prov:label` renders as inert escaped text (no forged node, no `URL` attribute) — proven in `packages/tsprov-render-dot/src/dot.test.ts`. |
